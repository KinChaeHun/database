데이터베이스 설계 및 구현 프로젝트

편의점 운영 데이터베이스 구축
작성자: 김채훈 (202532980)

1. 프로젝트 개요

이 프로젝트는 편의점 운영 과정에서 발생하는 상품 관리, 판매 관리, 폐기 관리, 직원 관리, 결제 관리 데이터를 체계적으로 다룰 수 있는 데이터베이스를 설계하고 구현하는 것을 목표로 한다.
개념적 설계부터 논리적·물리적 설계, SQL 구현까지 전 과정을 포함한다.

2. 요구사항 분석 요약
영역	주요 요구사항
상품 관리 : 상품코드, 상품명, 가격, 카테고리, 유통기한, 행사여부, 재고수량 저장 및 조회
판매 관리 : 판매번호, 판매일자, 직원, 결제방식 기록 및 조회
판매 상세	: 판매번호 + 상품코드 기준 판매수량 저장
폐기 관리	: 상품코드 + 폐기일자 기준 폐기수량 저장
직원 관리	: 직원의 기본 인적사항 관리
결제 관리	: 결제번호, 판매번호, 결제방식 저장

4. 개념적 설계 (ERD)

ERD는 개념적 모델링을 위해 엔터티와 관계를 시각적으로 표현한 다이어그램이다.
본 프로젝트에서는 다음 6개의 엔터티를 기반으로 구성하였다

상품 (PRODUCT)
판매 (SALE)
판매상세 (SALE_DETAIL)
폐기 (DISPOSAL)
직원 (EMPLOYEE)
결제 (PAYMENT)
ERD 파일: ERD.drawio

4. 논리적 설계 (관계형 스키마)

논리적 설계 단계에서는 각 엔터티를 관계형 테이블로 변환하고, 속성, 키, 제약조건을 정의하였다.
모든 테이블은 제3정규형(3NF)을 만족한다.

테이블 목록:
PRODUCT(상품)
SALE(판매)
SALE_DETAIL(판매상세)
DISPOSAL(폐기)
EMPLOYEE(직원)
PAYMENT(결제)

5. 물리적 설계

Oracle 기반으로 자료형, 제약조건, 인덱스 전략을 고려하여 테이블을 구현하였다.

인덱스 선정 이유

상품명(product_name): 상품 검색 빈도가 높음
판매일자(sale_date): 기간별 매출 조회
직원(employee_name): 특정 직원 판매 조회
상품코드(product_code): 판매 상세 조인 성능 향상
연락처(phone): 유니크한 검색 키
판매번호(sale_id): 결제 조인 성능 향상
역정규화는 필요하지 않다고 판단하였으며, 현재 구조는 무결성과 성능 모두 안정적이다.
